

## What Makes Window Functions Different?

Unlike `GROUP BY`, which collapses multiple rows into a single output row per group, **window functions** perform calculations across a set of rows (“window”) related to the current row, **without losing individual rows**.

Think of it as:  
- `GROUP BY` = make groups, one result per group.  
- Window function = keep every row, add a calculated column based on a sliding or partitioned frame.

---

## 1. PARTITION BY – Defining the “Window” Groups

### Purpose
Splits rows into logical groups (partitions). The window function resets for each partition. If you omit `PARTITION BY`, the whole result set is one partition.

### Syntax
```sql
function() OVER (PARTITION BY column1, column2 ORDER BY column3)
```

### Example Without/With Partition

```sql
SELECT 
    department,
    employee_name,
    salary,
    AVG(salary) OVER () AS company_avg,           -- no partition: one value for all rows
    AVG(salary) OVER (PARTITION BY department) AS dept_avg
FROM employees;
```

**Result (conceptual):**

| department | employee | salary | company_avg | dept_avg |
|------------|----------|--------|-------------|----------|
| Sales      | Alice    | 90000  | 78000       | 85000    |
| Sales      | Bob      | 80000  | 78000       | 85000    |
| Engineering| Carol    | 120000 | 78000       | 110000   |
| Engineering| Dave     | 100000 | 78000       | 110000   |

- `company_avg` repeats the same number (avg of all salaries).
- `dept_avg` resets per department.

### Common Pitfall
Forgetting `PARTITION BY` when you meant to have per-group calculations – you’ll get a constant across all rows.

## 2. ROW_NUMBER() – Unique Sequential Number per Partition

### Purpose
Assigns a unique sequential integer to each row **within a partition**, starting at 1. Ties get different numbers arbitrarily (unless you use `ORDER BY` to break ties).

### Syntax
```sql
ROW_NUMBER() OVER (PARTITION BY col1 ORDER BY col2)
```

### Example: Find the first order per customer
```sql
SELECT 
    customer_id,
    order_date,
    amount,
    ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date) AS order_sequence
FROM orders;
```

| customer_id | order_date | amount | order_sequence |
|-------------|------------|--------|----------------|
| 101         | 2025-01-10 | 100    | 1              |
| 101         | 2025-01-15 | 50     | 2              |
| 101         | 2025-01-20 | 75     | 3              |
| 102         | 2025-01-12 | 200    | 1              |

### Use Case: Deduplication
Delete duplicate rows (keep the first one per group):
```sql
WITH ranked AS (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id, product_id ORDER BY created_at) rn
    FROM purchases
)
DELETE FROM purchases
WHERE (user_id, product_id, created_at) IN (
    SELECT user_id, product_id, created_at FROM ranked WHERE rn > 1
);
```

### Important
- **No two rows get the same number** within a partition, even if `ORDER BY` columns have ties. The database decides the tie-breaker arbitrarily unless you add more `ORDER BY` columns.
- Use `ROW_NUMBER()` when you need a **strict unique ranking** (e.g., pagination, top-1 per group).

## 3. RANK() – Ranking with Gaps on Ties

### Purpose
Assigns a rank to each row. Ties get the same rank, and the next rank **skips** numbers (gaps).

### Comparison Table

| Value | ROW_NUMBER | RANK | DENSE_RANK |
|-------|------------|------|------------|
| 100   | 1          | 1    | 1          |
| 100   | 2          | 1    | 1          |
| 90    | 3          | 3    | 2          |
| 90    | 4          | 3    | 2          |
| 80    | 5          | 5    | 3          |

### Example: Sales competition
```sql
SELECT 
    salesperson,
    revenue,
    RANK() OVER (ORDER BY revenue DESC) AS rank_with_gaps,
    DENSE_RANK() OVER (ORDER BY revenue DESC) AS rank_no_gaps
FROM sales;
```

### When to Use Which
- `ROW_NUMBER()` – every row is distinct (e.g., sequential invoice lines).
- `RANK()` – you want ties to share rank, and you want the next rank to reflect how many people are ahead (classic competition ranking).
- `DENSE_RANK()` – you want ties to share rank, but ranks are consecutive (e.g., top 3 without gaps).

### Pitfall
If you forget `ORDER BY` inside `OVER()`, all rows get rank 1. That’s usually not what you want.

## 4. LEAD() and LAG() – Accessing Adjacent Rows

### Purpose
- `LAG(column, offset, default)` – get value from a row **before** the current row.
- `LEAD(column, offset, default)` – get value from a row **after** the current row.

Extremely useful for comparing a row with its neighbors (e.g., month-over-month change, time difference).

### Syntax
```sql
LAG(column_name, 1, 0) OVER (PARTITION BY group_col ORDER BY time_col)
LEAD(column_name, 1, NULL) OVER (PARTITION BY group_col ORDER BY time_col)
```

### Example: Calculate revenue change from previous month

```sql
SELECT 
    year,
    month,
    revenue,
    LAG(revenue, 1, 0) OVER (ORDER BY year, month) AS prev_month_revenue,
    revenue - LAG(revenue, 1, 0) OVER (ORDER BY year, month) AS change
FROM monthly_sales;
```

| year | month | revenue | prev_month_revenue | change |
|------|-------|---------|--------------------|--------|
| 2024 | 1     | 1000    | 0                  | 1000   |
| 2024 | 2     | 1100    | 1000               | 100    |
| 2024 | 3     | 1050    | 1100               | -50    |

### Example: Next order date per customer (using LEAD)
```sql
SELECT 
    customer_id,
    order_date,
    LEAD(order_date, 1) OVER (PARTITION BY customer_id ORDER BY order_date) AS next_order_date
FROM orders;
```

### Common Use Cases
- **Lead/Lag with default** – use `COALESCE` or the third parameter to handle `NULL` at boundaries.
- **Difference between rows** – combine `LAG` with subtraction.
- **Percentage change** – `(revenue - LAG(revenue)) / LAG(revenue) * 100`.

### Pitfall
Without `PARTITION BY`, `LAG` will look at the previous row in the entire result set, even across unrelated groups. Always ask: “Do I need to reset this per group?”

## 5. Combining Window Functions in One Query

You can have multiple window functions with different `OVER()` clauses. They are computed independently.

### Example
```sql
SELECT 
    employee_id,
    department,
    salary,
    ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rank_in_dept,
    AVG(salary) OVER (PARTITION BY department) AS dept_avg,
    salary - AVG(salary) OVER (PARTITION BY department) AS diff_from_dept_avg,
    LAG(salary, 1) OVER (PARTITION BY department ORDER BY salary) AS prev_salary_in_dept
FROM employees;
```

## Execution Order (Crucial)

Window functions are evaluated **after `WHERE`, `GROUP BY`, `HAVING`** but **before `ORDER BY`** of the main query.

Logical order:
```
FROM → WHERE → GROUP BY → HAVING → WINDOW FUNCTIONS → SELECT → DISTINCT → ORDER BY → LIMIT
```

This means:
- You can filter rows with `WHERE` before applying window functions.
- You cannot use a window function inside a `WHERE` clause directly (use a subquery/CTE).
- You can alias a window function in `SELECT` and then use that alias in `ORDER BY` (because `ORDER BY` runs after).

### Example of correct filtering after window function
```sql
WITH ranked AS (
    SELECT 
        customer_id,
        order_total,
        ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_total DESC) as rn
    FROM orders
    WHERE order_date >= '2025-01-01'
)
SELECT * FROM ranked WHERE rn = 1;
```

## Performance Considerations (Real‑world advice)

1. **Window functions are generally efficient** – they scan the data once and compute all frames in memory/disk order. But they can be heavy on large partitions because the database may need to sort each partition.

2. **Index the `PARTITION BY` and `ORDER BY` columns** – the database still needs to sort, but a matching index can avoid an explicit sort.

3. **Avoid redundant `OVER()` definitions** – use the `WINDOW` clause (standard SQL) to name a window and reuse it.

```sql
SELECT 
    salary,
    ROW_NUMBER() OVER w AS rn,
    AVG(salary) OVER w AS avg_salary
FROM employees
WINDOW w AS (PARTITION BY department ORDER BY salary);
```

4. **Be careful with large frames** – if you use `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` (the default for some aggregates like `SUM` over partition), the database may need to hold the entire partition in memory.

## Quick Reference Table

| Function | Description | With Ties | Unique per row? | Gaps? |
|----------|-------------|-----------|-----------------|-------|
| `ROW_NUMBER()` | Sequential number | No (arbitrary tie‑break) | Yes | N/A |
| `RANK()` | Rank with gaps | Same rank | No | Yes |
| `DENSE_RANK()` | Rank without gaps | Same rank | No | No |
| `LAG()` | Previous row’s value | N/A | N/A | N/A |
| `LEAD()` | Next row’s value | N/A | N/A | N/A |

| Clause | Purpose |
|--------|---------|
| `PARTITION BY` | Define groups for resetting the function |
| `ORDER BY` inside `OVER()` | Defines the order within each partition (required for ranking and lead/lag) |
| `ROWS BETWEEN` | Define a sliding frame (advanced, not covered here) |

---

## Final Pro Tips

1. **`ROW_NUMBER()` is your Swiss Army knife** – top‑N per group, pagination, deduplication.
2. **Don't confuse `RANK()` with `ROW_NUMBER()`** – test with duplicate values in your `ORDER BY` columns.
3. **Always use `PARTITION BY` unless you truly want the whole dataset** – forgetting it is a common bug.
4. **Lead/lag without explicit default** returns `NULL` at boundaries. Decide whether that’s acceptable or if you want `COALESCE`.
5. **Window functions are not allowed in `WHERE`** – use a CTE or subquery. That’s a frequent interview trap.
