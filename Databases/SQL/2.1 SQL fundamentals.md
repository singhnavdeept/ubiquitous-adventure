
## 1. SELECT – The Foundation of All Queries

### Purpose
Retrieve data from one or more tables. Think of it as a projection: you decide which columns and which rows.

### Basic Syntax
```sql
SELECT column1, column2, ...
FROM table_name
WHERE condition;   -- optional filtering
```

### Key Variations

| Clause / Option | Example | Explanation |
|----------------|---------|-------------|
| All columns | `SELECT * FROM users;` | Dangerous in production (performance, schema coupling). Use only for ad-hoc exploration. |
| Distinct rows | `SELECT DISTINCT country FROM users;` | Removes duplicates from result set. |
| Expressions | `SELECT price * quantity AS total FROM sales;` | Calculated columns. `AS` gives an alias. |
| Literals | `SELECT 'Customer: ' || name FROM users;` | Concatenation syntax depends on DB (e.g., `CONCAT()` in MySQL). |
| Limiting rows | `SELECT * FROM orders LIMIT 10;` | `LIMIT` (PostgreSQL, MySQL), `TOP` (SQL Server), `FETCH FIRST` (Oracle). |

### Common Pitfalls
- **`SELECT *` in application code** – leads to over-fetching, breaks when schema changes.
- **Missing `DISTINCT`** when you expect unique rows but get duplicates.
- **Not handling `NULL`** in expressions – any math with `NULL` yields `NULL`. Use `COALESCE()` or `ISNULL()`.

### Best Practice
```sql
-- Good: explicit columns, filter, limit
SELECT id, email, created_at
FROM users
WHERE active = true
ORDER BY created_at DESC
LIMIT 100;
```

## 2. JOIN – Combining Tables

### Why JOIN?
Relational databases normalize data across tables. JOIN reconstructs the logical view.

### Types of JOINs (Visual Summary)

```
INNER JOIN:        LEFT JOIN:          RIGHT JOIN:         FULL OUTER JOIN:
┌────┬────┐        ┌────┬────┐         ┌────┬────┐         ┌────┬────┐
│ ██  │    │        │ ██  │ ██ │         │    │ ██ │         │ ██  │ ██ │
│ ██  │    │        │ ██  │ ██ │         │    │ ██ │         │ ██  │ ██ │
└────┴────┘        └────┴────┘         └────┴────┘         └────┴────┘
 Only matches     All left + matches   All right + matches  All rows both
```

### Syntax and Example

**Tables:** `customers(id, name)`, `orders(id, customer_id, amount)`

```sql
-- INNER JOIN (most common)
SELECT c.name, o.amount
FROM customers c
INNER JOIN orders o ON c.id = o.customer_id;
-- Only customers who have placed an order.

-- LEFT JOIN
SELECT c.name, o.amount
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id;
-- All customers, with NULL for those without orders.

-- RIGHT JOIN (rare; can be rewritten as LEFT JOIN)
SELECT c.name, o.amount
FROM orders o
RIGHT JOIN customers c ON c.id = o.customer_id;  -- same as LEFT JOIN above

-- FULL OUTER JOIN
SELECT c.name, o.amount
FROM customers c
FULL OUTER JOIN orders o ON c.id = o.customer_id;
-- All customers and all orders, even if unmatched on either side.

-- CROSS JOIN (cartesian product – use sparingly)
SELECT c.name, o.amount
FROM customers c
CROSS JOIN orders o;
-- Every customer paired with every order. Dangerously large.
```

### Non-Equi JOINs
Join on conditions other than equality (e.g., ranges):
```sql
SELECT e.name, s.salary_grade
FROM employees e
JOIN salary_grades s ON e.salary BETWEEN s.min_salary AND s.max_salary;
```

### Self JOIN
Join a table to itself. Use different aliases.
```sql
SELECT e1.name AS employee, e2.name AS manager
FROM employees e1
LEFT JOIN employees e2 ON e1.manager_id = e2.id;
```

### Common Pitfalls
- **Forgetting JOIN condition** → produces a Cartesian product (CROSS JOIN), bringing down performance.
- **Using `LEFT JOIN` but filtering on right table in `WHERE`** → turns it into an `INNER JOIN` (because `NULL` fails the condition). Move such filters to the `ON` clause if you want to keep left rows.
- **Duplicate rows** – if the right table has multiple matches, rows multiply. Understand your data cardinality.

### Best Practices
- Always qualify column names with table aliases (e.g., `c.id` not `id`).
- Prefer `INNER JOIN` unless you explicitly need unmatched rows.
- Test with small `LIMIT` first when joining large tables.

## 3. GROUP BY – Aggregating Rows

### Purpose
Group rows that have the same values in specified columns, then apply aggregate functions (e.g., `SUM`, `COUNT`, `AVG`, `MAX`, `MIN`) to each group.

### Syntax
```sql
SELECT column1, AGGREGATE_FUNCTION(column2)
FROM table
WHERE condition
GROUP BY column1;
```

### Example
```sql
-- Total sales per customer
SELECT customer_id, SUM(amount) AS total_spent
FROM orders
GROUP BY customer_id;

-- With multiple grouping columns
SELECT customer_id, DATE(created_at) AS order_date, COUNT(*) AS orders_per_day
FROM orders
GROUP BY customer_id, DATE(created_at);
```

### The Rule (Very Important)
**Every column in the `SELECT` that is not an aggregate function must appear in the `GROUP BY` clause.**

❌ Wrong:
```sql
SELECT customer_id, status, SUM(amount)
FROM orders
GROUP BY customer_id;   -- 'status' is missing from GROUP BY
```

✅ Correct:
```sql
SELECT customer_id, status, SUM(amount)
FROM orders
GROUP BY customer_id, status;
```

### Aggregate Functions Cheat Sheet

| Function | Description | NULL handling |
|----------|-------------|----------------|
| `COUNT(*)` | Number of rows in group | Counts all rows |
| `COUNT(col)` | Number of non‑NULL values in col | Ignores NULLs |
| `SUM(col)` | Sum of non‑NULL values | Ignores NULLs |
| `AVG(col)` | Average of non‑NULL values | Ignores NULLs |
| `MAX(col)` / `MIN(col)` | Highest/lowest value | Ignores NULLs |

### Pitfalls
- **Forgetting `GROUP BY`** when using aggregates without it – you get one row for the whole table, which may be intentional but often a bug.
- **Including non‑aggregated, non‑grouped columns** – the database will either error (strict mode) or give arbitrary values (non‑strict). Don't rely on that.
- **`COUNT(*)` vs `COUNT(column)`** – `COUNT(*)` includes rows where all columns are NULL; `COUNT(column)` excludes NULLs. Understand the difference.

## 4. ORDER BY – Sorting Results

### Purpose
Sort the result set by one or more columns, ascending (`ASC`) or descending (`DESC`). Applies **after** all other operations (`WHERE`, `GROUP BY`, `HAVING`), but **before** `LIMIT`.

### Syntax
```sql
SELECT column1, column2
FROM table
ORDER BY column1 [ASC|DESC], column2 [ASC|DESC], ...;
```

### Examples
```sql
-- Single column, descending
SELECT name, salary
FROM employees
ORDER BY salary DESC;

-- Multiple columns: first by department ascending, then by salary descending
SELECT department, name, salary
FROM employees
ORDER BY department ASC, salary DESC;

-- Order by expression or alias
SELECT name, salary * 1.1 AS bonus
FROM employees
ORDER BY bonus DESC;

-- Order by column position (not recommended for maintenance)
SELECT name, salary
FROM employees
ORDER BY 2 DESC;   -- 2 refers to the second column (salary)
```

### NULL Sorting Behavior (Database‑specific)

| Database | `ASC` NULLS | `DESC` NULLS |
|----------|-------------|---------------|
| PostgreSQL | NULLS LAST (default) | NULLS FIRST |
| MySQL | NULLS FIRST (default) | NULLS LAST |
| SQL Server | NULLS FIRST | NULLS LAST |

To be explicit (PostgreSQL, Oracle, SQLite):
```sql
ORDER BY salary ASC NULLS LAST;
```

### Pitfalls
- **Order of operations**: `ORDER BY` happens after `SELECT`, so you can use column aliases. But it happens before `LIMIT`, so sorting a huge table then taking top 10 is expensive – use indexes to avoid.
- **Non‑deterministic ordering** – if you `ORDER BY` a non‑unique column, rows with equal values can appear in any order between runs. Add a unique column as tie‑breaker.

### Best Practice
```sql
ORDER BY created_at DESC, id DESC   -- unique tie‑breaker
```

## 5. HAVING – Filtering After Aggregation

### Purpose
`WHERE` filters rows **before** grouping. `HAVING` filters groups **after** grouping. You cannot use aggregate functions in `WHERE`, but you can in `HAVING`.

### Syntax
```sql
SELECT column1, AGG(column2)
FROM table
WHERE condition               -- filters rows pre‑aggregation
GROUP BY column1
HAVING AGG(column2) condition; -- filters groups post‑aggregation
```

### Example
```sql
-- Find customers who have spent more than $1000 total
SELECT customer_id, SUM(amount) AS total_spent
FROM orders
WHERE status != 'cancelled'   -- exclude cancelled orders before summing
GROUP BY customer_id
HAVING SUM(amount) > 1000;
```

### Comparison: WHERE vs HAVING

| Clause | When applied | Can use aggregates | Can use aliases |
|--------|--------------|--------------------|-----------------|
| `WHERE` | Before grouping | ❌ No | ❌ No (in most DBs) |
| `HAVING` | After grouping | ✅ Yes | ✅ Yes (some DBs) |

### Common Mistake
```sql
-- Wrong: Using aggregate in WHERE
SELECT department, AVG(salary)
FROM employees
WHERE AVG(salary) > 50000   -- ERROR
GROUP BY department;

-- Correct: Use HAVING
SELECT department, AVG(salary)
FROM employees
GROUP BY department
HAVING AVG(salary) > 50000;
```

### Tip
Often you can move a condition from `HAVING` to `WHERE` to improve performance. Filter as early as possible.

## Putting It All Together: Execution Order (Mental Model)

Despite the way we write SQL, the database engine logically executes clauses in this order:

```
1. FROM / JOIN         – Identify tables and join them.
2. WHERE               – Filter rows based on conditions.
3. GROUP BY            – Group rows into aggregates.
4. HAVING              – Filter groups.
5. SELECT              – Evaluate expressions, aliases, DISTINCT.
6. ORDER BY            – Sort the final result.
7. LIMIT / OFFSET      – Limit rows returned.
```

### Example combining all concepts

```sql
-- For each product category with more than 5 completed orders,
-- show the category and total revenue, sorted by revenue descending,
-- only the top 3 categories.
SELECT 
    p.category,
    SUM(o.amount) AS total_revenue
FROM orders o
JOIN products p ON o.product_id = p.id
WHERE o.status = 'completed'        -- 1. Filter rows
GROUP BY p.category                 -- 2. Group
HAVING COUNT(*) > 5                 -- 3. Filter groups
ORDER BY total_revenue DESC         -- 4. Sort
LIMIT 3;                            -- 5. Limit
```

| `SELECT` | Choose columns | 5 | Yes (if in group) |

| `WHERE` | Filter rows | 2 | No |
| `GROUP BY` | Create groups | 3 | No |
| `HAVING` | Filter groups | 4 | Yes |
| `ORDER BY` | Sort final rows | 6 | Yes |
| `LIMIT` | Restrict rows | 7 | No |

## Final Pro Tips from the Field

1. **Always format your SQL** – readability saves debugging hours.
2. **Use meaningful aliases** – `total_revenue` not `sum`.
3. **Prefer `INNER JOIN` unless you need `LEFT`** – it signals intent and often performs better.
4. **Test aggregates with small samples** – add `LIMIT 100` inside a subquery before joining to huge tables.
5. **Know your NULLs** – `NULL` is not equal to anything, not even `NULL`. Use `IS NULL` / `IS NOT NULL`.
6. **Index for `WHERE`, `JOIN`, `ORDER BY`** – indexes on columns used in these clauses dramatically speed up queries.
