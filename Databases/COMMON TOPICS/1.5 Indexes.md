# Comprehensive Engineering Notes: Indexes (A Deep Dive)

## Part 1: The Foundational Index Types

### 1.1 B-Tree (The Workhorse)
**The Default. The Reliable. The Overused.**

#### What It Actually Is (Not "Binary Tree")
A **Balanced Tree**. Specifically, a high-fanout tree where each node contains multiple keys and pointers to child nodes. "B" stands for **Bayer** (co-inventor) or **Balanced**—history is unclear, physics is not.

#### The Physics (Crucial for Performance Intuition)
- **Node Size = Page Size:** Each B-tree node is exactly the size of a database **page** (usually 8KB in PostgreSQL, 16KB in MySQL/InnoDB).
- **Fanout:** A single 8KB page can hold hundreds of index entries (keys + row pointers). This means a B-tree is **extremely shallow**.
    - A 1 million row table: Tree height = **3 levels**.
    - A 1 billion row table: Tree height = **4-5 levels**.
- **Traversal Cost:** Finding a specific key `WHERE id = 12345` requires reading that many pages from disk (or cache). **3-4 I/O operations.** That's why B-tree lookups are O(log n) but with a tiny constant factor.

#### The Two Variants (Understand the Difference)

| Feature                | **Primary (Clustered) Index** (MySQL InnoDB, SQL Server) | **Secondary Index** (PostgreSQL Heap, MySQL Non-PK) |
| :--------------------- | :------------------------------------------------------- | :-------------------------------------------------- |
| **Leaf Node Contains** | **The Entire Row Data**                                  | **The Indexed Columns + Primary Key Pointer**       |
| **Lookup Path**        | Index Leaf -> Done (1 hop)                               | Index Leaf -> **Heap Fetch** (2 hops)               |
| **Implication**        | Primary Key size affects **every** secondary index.      | Heap fetches cause random I/O.                      |

**PostgreSQL Note:** PostgreSQL **never** has clustered indexes in the InnoDB sense. It uses **Heap-Only Tuples (HOT)** optimization to avoid index bloat, but the row data lives separately from indexes.

#### The Hidden Cost: Write Amplification
An `INSERT` on a table with 5 B-tree indexes requires:
1. Insert row into heap.
2. Write to Index A leaf page (possibly causing a **page split**).
3. Write to Index B leaf page.
4. Write to Index C leaf page.
5. Write to Index D leaf page.
6. Write to Index E leaf page.

**Rule of Thumb:** Every additional index adds **~10-20% overhead** to write latency and WAL volume. Indexes are **read accelerators** paid for with **write penalties**.

#### When to Use B-Tree
- **Equality:** `WHERE id = 5`
- **Range Scans:** `WHERE created_at BETWEEN '2026-01-01' AND '2026-02-01'`
- **Sorting:** `ORDER BY score DESC` (if index matches sort order)
- **LIKE Prefix:** `WHERE name LIKE 'Joh%'` (anchored at start)

### 1.2 Hash Index
**The Specialist. Laser Focused. Zero Flexibility.**

#### What It Is
A classic in-memory hash table persisted to disk. Key -> Hash Function -> Bucket -> Row Pointer.

#### The Physics
- **Access Pattern:** **O(1) single-key lookup**. No tree traversal.
- **Concurrency (PostgreSQL):** Prior to PostgreSQL v10, Hash indexes were **not WAL-logged**. A crash meant the index was corrupt and needed `REINDEX`. **Never use Hash in PG < 10.** Since PG 11, they are crash-safe and size-efficient (smaller than B-tree).

#### The Brutal Limitations
- **No Range Scans:** `WHERE id > 5` **cannot use a Hash index**. The hashing destroys order.
- **No Sorting:** `ORDER BY id` **cannot use a Hash index**.
- **No Partial Matching:** `WHERE name LIKE 'Joh%'` **cannot use a Hash index**.

#### When to Use Hash (The Rare Case)
You have a very large, static lookup table where queries are **always** exact match `WHERE key = ?` and you want to shave ~20% off index size compared to B-tree. Think: **Country codes**, **Currency symbols**, **UUID key mappings**.

**Verdict:** You will almost never need to reach for this explicitly. B-tree is good enough for equality.

### 1.3 GiST (Generalized Search Tree)
**The Extensible Framework. Geometry. Full-Text. The Unexpected.**

#### What It Is
Not a single algorithm. GiST is a **template** for building balanced tree indexes for **complex data types**. PostgreSQL uses it as the foundation for **Geometric Indexing** (PostGIS) and **Full-Text Search** (tsvector).

#### The Physics of a GiST Over Geometry
Imagine a B-tree, but the keys aren't numbers—they are **bounding boxes** (rectangles that contain the shape).
- **Root Node:** Contains a bounding box that covers **all geometries in the table**.
- **Child Nodes:** Contain progressively smaller bounding boxes.
- **Leaf Nodes:** Contain the actual geometry + row pointer.

**Query: "Find all parks within 1km of this point."**
1. Start at root: "Is the point inside the root bounding box? Yes."
2. Traverse children: "Is point inside child A's box? No. Child B's box? Yes."
3. Prune massive subtrees instantly. Only compute exact distance on the few geometries that survived the box filter.

#### Use Cases for GiST
- **PostGIS:** `WHERE ST_DWithin(geom, point, 1000)` -> GiST is mandatory for performance.
- **Full-Text Search:** `WHERE to_tsvector('english', body) @@ to_tsquery('database & index')` -> GiST or GIN.
- **Range Types:** `WHERE valid_period @> CURRENT_DATE` -> GiST indexes on `daterange` columns.

#### GiST vs. GIN (The Trade-Off)
| Feature | GiST | GIN |
| :--- | :--- | :--- |
| **Write Speed** | Faster (updates are local to the tree node) | Slower (must update multiple posting lists) |
| **Read Speed** | Slower (scans may touch more false positives) | Faster (inverted index is extremely precise) |
| **Size** | Smaller | Larger (often 2-3x GiST) |

**Engineering Rule:** Use **GIN** for text search if the data is **read-heavy and rarely updated** (e.g., product catalog). Use **GiST** for text search if the data is **frequently updated** (e.g., live comment feeds) or for spatial data.

### 1.4 GIN (Generalized Inverted Index)
**The "Google Inside Your Database" Index**

#### What It Is
An **Inverted Index**. Instead of "Key -> Row", it's **"Token -> List of Row IDs"**.

#### The Physics of a GIN Index on a JSONB Column
```json
{"tags": ["postgres", "database", "tuning"]}
```
**GIN Build Process:**
1. Extract all keys and values (the **tokens**).
2. For token `"postgres"`, add Row ID 123 to its **posting list**.
3. For token `"database"`, add Row ID 123 to its posting list.

**Query: `WHERE payload @> '{"tags": ["database"]}'`**
1. Look up token `"database"`.
2. Fetch the **posting list** (which might be a sorted array of Row IDs).
3. Return rows. Done.

#### The Critical Feature: Fast AND/OR Operations
Since posting lists are sorted arrays of Row IDs, GIN can perform **Boolean Intersection** (`AND`) and **Union** (`OR`) in **streaming merge time**.
- `WHERE payload @> '{"tags": ["postgres"]}' AND payload @> '{"tags": ["tuning"]}'`
- GIN fetches posting list for "postgres" (IDs: 1, 5, 10).
- GIN fetches posting list for "tuning" (IDs: 5, 10, 15).
- **Merge algorithm:** Walk both sorted lists. Match: ID 5 and 10.
- **Result:** Instant intersection without scanning a single row of table data.

#### The Write Penalty (Significant)
GIN indexes use a **Fast Update** list to avoid updating the main index structure on every insert. This list is flushed periodically (`gin_clean_pending_list`).
- **Consequence:** A GIN index with a large pending list is **slow to read** because it must scan both the main index *and* the pending list.
- **Consequence:** `INSERT` into a heavily GIN-indexed table eventually triggers background autovacuum work.

#### Use Cases
- **JSONB `@>` queries** (as above).
- **Array overlap:** `WHERE tags && ARRAY['urgent', 'important']`.
- **Full-Text Search:** `WHERE tsvector @@ to_tsquery('query')`. (Actually, GIN is preferred over GiST for static text).

## Part 2: Indexing Strategies (The Art of Column Order)

### 2.1 Composite Indexes (Multi-Column B-Tree)
**The Most Powerful Tool Most Developers Misuse**

#### The Golden Rule: The "Phone Book" Analogy
A composite index on `(last_name, first_name)` sorts data **first** by `last_name`, then **within groups of identical last names** by `first_name`.

#### Query Eligibility (The Leftmost Prefix Rule)

**Index:** `CREATE INDEX idx_name ON users (last_name, first_name, middle_initial);`

| Query WHERE Clause | Uses Index? | Why |
| :--- | :--- | :--- |
| `WHERE last_name = 'Smith'` | **YES** | Matches prefix. Index scans all 'Smith' entries efficiently. |
| `WHERE last_name = 'Smith' AND first_name = 'John'` | **YES** | Perfect match on first two columns. |
| `WHERE first_name = 'John'` | **NO** | Skipped the first column. Index is useless for this. (Think: Can you look up "John" in a phone book sorted by last name? No.) |
| `WHERE last_name LIKE 'Sm%' AND first_name = 'John'` | **Partial** | Scans range for 'Sm%', then filters `first_name`. |
| `WHERE last_name = 'Smith' AND middle_initial = 'A'` | **Partial** | Uses `last_name` only. The skip over `first_name` breaks index use for `middle_initial`. |

#### The Engineering Decision: Column Order
**Rule:** Order columns in the index from **Most Selective** to **Least Selective**, **BUT** prioritize columns used in **Equality** over **Range**.

**Scenario:**
- Query: `WHERE status = 'active' AND created_at > '2026-01-01'`
- Data: `status` has only 3 values (low selectivity). `created_at` is unique per second (high selectivity).

**Option A: Index `(status, created_at)`**
- Index finds all 'active' rows (which is 30% of the table), *then* filters by date.
- **Verdict: Good.** Database uses index for both parts.

**Option B: Index `(created_at, status)`**
- Index finds date range efficiently.
- **But** the `status` column is now **not contiguous** in the index. The database must filter out non-active rows one by one.
- **Verdict: Bad.** Less efficient.

**The Correct Order: Equality Columns First, Range Column Last.**
`(status, created_at)` is the correct choice here.

### 2.2 Covering Indexes (The Ultimate Optimization)
**"Index-Only Scan" / "Covering Index"**

#### The Problem: The Heap Fetch
Standard B-tree lookup (in PostgreSQL or MySQL Secondary Index):
1. Search Index for key. -> Finds Row Pointer (e.g., Primary Key value).
2. **Go to the Heap/Clustered Index** to fetch the actual row data (`SELECT *`).
3. That second step is **Random I/O** (expensive).

#### The Solution: INCLUDE Clause (PostgreSQL 11+ / SQL Server)
```sql
CREATE INDEX idx_covering ON orders (customer_id) INCLUDE (order_date, total_amount);
```
**What happens now:**
- The index leaf node stores `customer_id` (for searching) **PLUS** a copy of `order_date` and `total_amount`.
- Query: `SELECT order_date, total_amount FROM orders WHERE customer_id = 123;`
- **Result:** The database **never touches the main table**. It reads only the index. This is an **Index-Only Scan**.

#### The Trade-Off
- **Benefit:** Query is **2x-10x faster** (avoids random I/O).
- **Cost:** Index is **larger** on disk (duplicates data from the heap).
- **Cost:** Writes are **slower** (must update the index copies as well as the heap).

#### The Pragmatic Pattern
You don't need `INCLUDE` for **Primary Key lookups** in PostgreSQL if the table has low update frequency. But for **secondary indexes**, adding 1-2 frequently selected columns via `INCLUDE` is often the difference between a 200ms query and a 2ms query.

#### A Note for MySQL Users
MySQL InnoDB **automatically** includes the **Primary Key** in every secondary index. That means `INDEX (customer_id)` is inherently covering for queries like `SELECT id FROM orders WHERE customer_id = 123`. To cover other columns, you must use a **Composite Index** `INDEX (customer_id, order_date)` (no separate `INCLUDE` syntax).

## Part 3: The Index Dark Arts (What They Don't Teach in Tutorials)

### 3.1 Index Bloat (The Silent Killer)
B-tree pages split when full. In PostgreSQL, an `UPDATE` creates a **new row version** (due to MVCC). The old row version remains in the index until `VACUUM` cleans it up.
- **Symptom:** Table size is 10GB. Index size is 15GB. Queries are slow despite "using index."
- **Fix:** `REINDEX TABLE CONCURRENTLY` (PostgreSQL) or `OPTIMIZE TABLE` (MySQL).

### 3.2 Correlation (Why `ORDER BY` Matters)
PostgreSQL's query planner calculates **correlation** between index order and physical row order.
- If correlation is near **1.0**: Index scan is fast (rows are sequential on disk).
- If correlation is near **0.0**: Planner may choose a **Bitmap Heap Scan** over Index Scan to avoid random I/O.

### 3.3 Partial Indexes (Indexing a Subset)
```sql
CREATE INDEX idx_active_users ON users (last_login) WHERE status = 'active';
```
- **Size:** Tiny (only indexes 10% of the table).
- **Query Match:** `SELECT * FROM users WHERE status = 'active' AND last_login > '2026-01-01';` -> Lightning fast.
- **Maintenance:** `INSERT` into `users` with `status='inactive'` **does not update this index**. Zero write penalty.

### 3.4 Expression Indexes (Indexing Computed Values)
```sql
CREATE INDEX idx_lower_email ON users (LOWER(email));
```
- **Query:** `SELECT * FROM users WHERE LOWER(email) = 'john@doe.com';`
- **Benefit:** Case-insensitive search without storing a separate column.

## Summary Matrix: Index Type Selection Guide

| If your query does this... | Use this Index Type |
| :--- | :--- |
| `WHERE x = 5` or `BETWEEN` | **B-Tree** |
| `WHERE point <-> geometry < 100` | **GiST** (PostGIS) |
| `WHERE jsonb_col @> '{"key":"val"}'` | **GIN** |
| `WHERE array_col && ARRAY[1,2]` | **GIN** |
| `WHERE tsvector @@ to_tsquery('word')` | **GIN** (or GiST for heavy writes) |
| Exact match on huge static lookup | **Hash** (rarely) |
| Avoiding heap fetches | **Covering Index** (INCLUDE) |
| Filtering + Sorting | **Composite B-Tree** (order matters!) |
| Indexing only active rows | **Partial Index** |

**Final Engineering Wisdom:**
*Indexes are not a checklist item. They are a **cache of sorted data**. Every index you add is a promise to the database that you will ask questions in that exact order. If you stop asking those questions, the index becomes dead weight. Review your index usage quarterly with `pg_stat_user_indexes` and drop the ones with `idx_scan = 0`.*
