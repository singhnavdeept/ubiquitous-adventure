
# Comprehensive Engineering Notes: Constraints

## The Philosophy: Push Logic as Close to the Data as Possible
Every constraint you write in the database is a bug you **do not have to write in the application layer**. Application code changes, ORMs are bypassed by bulk imports, and developers make mistakes at 2 AM. The database is the **last line of defence**.

---

## 1. NOT NULL
**The Simplest, Most Violated Constraint**

### Syntax
```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    email TEXT NOT NULL,
    nickname TEXT -- Implicitly NULL allowed
);
```

### The Engineering Reality
- **Storage Overhead:** Zero. A `NULL` is indicated by a **bitmask** in the row header (the *null bitmap*). `NOT NULL` simply means the database doesn't check that bit; it trusts the value is there.
- **The Trap: Empty Strings (`''`).**
    - `NOT NULL` prevents `NULL`. It does **NOT** prevent `''` (empty string).
    - If you need to enforce "something must be there," you need a `CHECK` constraint (e.g., `CHECK (email <> '')`).
- **The Performance Win:** Queries with `WHERE column IS NOT NULL` on a nullable column can sometimes use indexes less efficiently. Defining the column as `NOT NULL` provides the planner with absolute certainty, leading to better query plans.

### Pro Tip: The Default Dilemma
Avoid this anti-pattern:
```sql
-- BAD: What does empty string mean? "I don't know" or "Not provided"?
middle_name TEXT NOT NULL DEFAULT '' 
```
If data is optional, **allow NULL**. `NULL` is explicit ambiguity. An empty string is accidental ambiguity.

## 2. UNIQUE Constraint
**Guaranteeing Distinctness**

### Syntax
```sql
-- Single Column
CREATE TABLE users (
    email TEXT UNIQUE
);

-- Composite (Powerful)
CREATE TABLE team_members (
    team_id INT,
    user_id INT,
    UNIQUE (team_id, user_id) -- A user can only be on a team once
);
```

### Under the Hood: The Index Link
A `UNIQUE` constraint is **physically enforced by a unique B-Tree index**. You cannot separate the two.
- **Implication:** Unique constraints make `SELECT` lookups on that column lightning fast (O(log n)).
- **Implication:** Unique constraints make `INSERT/UPDATE` slightly slower [[Dictionary#Slowing of writes due to indexing]](must probe the index to check for conflict).

### The NULL Loophole (Critical Knowledge)
In standard SQL, **`NULL` is not equal to `NULL`**.
Therefore, a `UNIQUE` column can contain **multiple NULL values**.

```sql
-- This is VALID in PostgreSQL and MySQL (if not in strict mode)
INSERT INTO users (email) VALUES (NULL), (NULL); 
-- Both rows inserted. The unique constraint ignores NULLs.
```
If you need "Unique if provided, but multiple NULLs allowed," this is perfect.
If you need "At most one NULL," you must use a **Partial Unique Index** (PostgreSQL magic):
```sql
CREATE UNIQUE INDEX unique_active_email ON users (email) WHERE email IS NOT NULL;
```

### The Concurrency Race Condition
You cannot safely enforce uniqueness in application code.
**Thread A:** `SELECT COUNT(*) WHERE email = 'x'` -> Returns 0.
**Thread B:** `SELECT COUNT(*) WHERE email = 'x'` -> Returns 0.
**Thread A:** `INSERT`.
**Thread B:** `INSERT`.
**Result:** Duplicate entry unless the **database constraint** steps in. Rely on the database.

## 3. PRIMARY KEY Constraint
**The Master Pointer**

### Definition
A `PRIMARY KEY` is logically equivalent to **`UNIQUE` + `NOT NULL`**. However, it carries special significance in relational theory and physical storage.

### Engineering Implications of Choosing a PK

#### a) Clustered Index (MySQL / SQL Server Default)
The Primary Key dictates the **physical order of rows on disk**.
- **Best Practice:** Use a strictly increasing value (e.g., `BIGSERIAL`, `UUIDv7`).
- **Consequence of UUIDv4:** Random UUIDs cause **page splits** (fragmentation) because new rows are inserted in the middle of the data file, not appended at the end. This destroys insert throughput and bloats the cache.

#### b) Heap Storage (PostgreSQL)
The Primary Key is just a unique index. Row location is separate.
- **Consequence:** All foreign key lookups require an **extra I/O hop** (Index -> Heap) compared to a clustered index.
- **Benefit:** Inserts are very fast regardless of PK randomness.

### The Composite Primary Key
```sql
CREATE TABLE order_items (
    order_id BIGINT,
    product_id BIGINT,
    PRIMARY KEY (order_id, product_id)
);
```
**When to use:** When the entity's identity is defined by its relationships (junction tables).
**When to avoid:** When you need to reference this row from another table (a 2-column FK is cumbersome). In that case, use a **Surrogate Key** (`id BIGSERIAL`) and put a `UNIQUE` constraint on `(order_id, product_id)`.

## 4. FOREIGN KEY Constraint
**The Keeper of Relationships**

### Syntax & Actions
```sql
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT REFERENCES users(id) ON DELETE CASCADE
);
```

### The Action Matrix (`ON DELETE` / `ON UPDATE`)

| Option | Behavior | Use Case |
| :--- | :--- | :--- |
| `RESTRICT` / `NO ACTION` | Blocks deletion of parent if child exists. | **Default & Safest.** |
| `CASCADE` | Delete parent -> Delete all children. | Comments on a deleted Post. |
| `SET NULL` | Delete parent -> Set child FK to `NULL`. | Unassigning a task when the assignee leaves. |
| `SET DEFAULT` | Delete parent -> Set child FK to default. | Rarely used. |

### The Performance Tax (Why Developers Remove Them in Production)
Foreign Keys are **not free**.
1.  **Write Penalty:** Every `INSERT` or `UPDATE` on the `orders` table must perform a read on the `users` table index to verify `user_id` exists. This adds latency.
2.  **Lock Contention:** Deleting a parent row (`users`) that has 10,000 child rows (`orders`) with `ON DELETE CASCADE` will hold a lock on the **child table** for the duration of the cascade, potentially blocking other writes.

**The Engineer's Middle Ground:**
- **Always enforce FKs in Dev/Staging.** They catch ORM mapping bugs instantly.
- **In High-Volume OLTP Production:** Consider removing FKs for **hot paths** (e.g., logging tables, event streams) where write throughput > 50k/sec. **BUT:** You **MUST** have a nightly reconciliation job that scans for orphans and alerts the on-call engineer. If you cannot afford that job, you cannot afford to drop the FK.

### The Surprising Use Case: Empty FK Column
A column can be `FOREIGN KEY ... REFERENCES ...` and also allow `NULL`.
This is **Semantically Perfect** for optional relationships.
```sql
manager_id BIGINT REFERENCES users(id) -- NULL means "No Manager"
```

## 5. CHECK Constraint
**The Custom Business Rule Enforcer**

### Syntax
```sql
CREATE TABLE products (
    price NUMERIC CHECK (price > 0),
    discount NUMERIC CHECK (discount >= 0 AND discount <= 100),
    status TEXT CHECK (status IN ('draft', 'published', 'archived'))
);
```

### The Power of Declarative Validation
A `CHECK` constraint evaluates a boolean expression. If `FALSE` or `NULL`, the insert/update fails.

**Advanced Pattern: Multi-Column Validation**
```sql
-- Ensure start_date is always before end_date
CONSTRAINT valid_date_range CHECK (start_date <= end_date)
```

**Advanced Pattern: Complex Logic**
```sql
-- Discount code required ONLY IF discount > 0
CONSTRAINT discount_code_required CHECK (
    (discount = 0 AND promo_code IS NULL) OR 
    (discount > 0 AND promo_code IS NOT NULL)
)
```

### The Hidden Danger: `CHECK` and Performance
- **Fast Validation:** Simple comparisons (`price > 0`) are near-zero cost.
- **Slow Validation:** `CHECK` constraints that call **User-Defined Functions (UDFs)** or subqueries.
    ```sql
    -- NEVER DO THIS (PostgreSQL)
    CHECK (validate_email_with_external_api(email))
    ```
    This makes every `INSERT` a network call. **Constraints must be deterministic and fast.**

## 6. Deferrable Constraints (The Advanced Tool)
**When "Transaction" Means "I'll Fix It Later"**

### The Problem
You have two rows that mutually reference each other (a circular FK).
```sql
-- Table A references B. Table B references A.
INSERT INTO A (id, b_id) VALUES (1, 2); -- Fails because B:2 doesn't exist yet.
INSERT INTO B (id, a_id) VALUES (2, 1); -- Fails because A:1 doesn't exist yet.
```

### The Solution: `DEFERRABLE INITIALLY DEFERRED`
This tells the database: *"Check this constraint only at the **end** of the transaction, not after every single row."*

```sql
CREATE TABLE a (
    id INT PRIMARY KEY,
    b_id INT REFERENCES b(id) DEFERRABLE INITIALLY DEFERRED
);
```

Now, inside a single `BEGIN; ... COMMIT;` block, you can insert A and B in any order, and they will only validate at `COMMIT` time.

**Engineering Note:** This is rare outside of complex graph structures or tree closures. Use it sparingly; it makes reasoning about transaction state harder.

## Engineer's Cheat Sheet: Constraint Priority

When designing a schema, apply constraints in this order of importance:

1.  **PRIMARY KEY** (Establish identity).
2.  **NOT NULL** (Eliminate ambiguity where data is mandatory).
3.  **FOREIGN KEY** (Prevent orphaned data).
4.  **UNIQUE** (Prevent duplicates).
5.  **CHECK** (Enforce domain rules).

**If you remember only one thing from this document:**
> *It is **100x easier** to remove a constraint that is hurting performance than it is to **clean up data** that was allowed to become corrupt because a constraint was missing.*
