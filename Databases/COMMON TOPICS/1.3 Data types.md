# Comprehensive Engineering Notes: Data Types & Storage

## 1. Numeric Types
**The Spectrum from Exact Math to Approximate Speed**

### Core Variants

| Type | PostgreSQL | MySQL | SQL Server | Storage | Use Case |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Integer (Small)** | `SMALLINT` | `SMALLINT` | `SMALLINT` | 2 bytes | Age, Year, Status Flags |
| **Integer (Standard)** | `INTEGER` | `INT` | `INT` | 4 bytes | User ID, Quantity, Foreign Keys |
| **Integer (Big)** | `BIGINT` | `BIGINT` | `BIGINT` | 8 bytes | Event Counters, High-Volume IDs |
| **Arbitrary Precision** | `NUMERIC(p,s)` | `DECIMAL(p,s)` | `DECIMAL(p,s)` | Variable | **Money, Financial Calculations** |
| **Floating Point** | `REAL` / `DOUBLE` | `FLOAT` / `DOUBLE` | `FLOAT` / `REAL` | 4/8 bytes | Sensor Data, Scientific, Geolocation |

### The Critical Engineering Distinction: Exact vs. Approximate

**`NUMERIC` / `DECIMAL` (Exact Math)**
- **Physics:** Stored as a packed string of digits with a scale pointer.
- **Behaviour:** `0.1 + 0.2 = 0.3`. Always.
- **The Catch:** **CPU Intensive**. Arithmetic operations on `NUMERIC` are software-emulated; they are roughly **10x-50x slower** than integer math. Index scans on `NUMERIC` are also slower due to byte comparison complexity.

**`FLOAT` / `REAL` (IEEE 754 Approximate Math)**
- **Physics:** Binary approximation. `0.1` cannot be represented exactly in binary (it's a repeating fraction like 1/3 in decimal).
- **Behaviour:** `0.1::FLOAT + 0.2::FLOAT = 0.30000000000000004`.
- **Golden Rule for Finance:** **NEVER USE FLOAT FOR MONEY.** Use `NUMERIC(19,4)` or store as **Integer Cents** (e.g., `$10.99` -> `1099`). This avoids rounding errors when summing millions of transactions.

### Auto-Incrementing Keys: The Modern Standard

- **Avoid `SERIAL` (Postgres) / `AUTO_INCREMENT` (MySQL).** They are 32-bit by default (2.1 billion limit). That sounds huge, but a viral social media app or a IoT sensor fleet will burn through that in months.
- **Use `BIGINT` with `IDENTITY` or `UUIDv7`.**
    - `BIGSERIAL` (Postgres): 64-bit.
    - `UUIDv7`: Sortable, unique across distributed systems without a central sequence generator. **Trade-off:** UUID is 16 bytes vs BIGINT 8 bytes -> larger indexes, slightly slower joins.

## 2. Text Types
**The Semantics of "What Does 'Empty' Mean?"**

### Core Variants & Storage

| Type | Standard SQL | PostgreSQL | MySQL | Description |
| :--- | :--- | :--- | :--- | :--- |
| **Fixed Length** | `CHAR(n)` | `CHAR(n)` | `CHAR(n)` | Pads with spaces. **Almost never used.** Wastes space. |
| **Variable Length** | `VARCHAR(n)` | `VARCHAR(n)` | `VARCHAR(n)` | Standard string with a length limit. |
| **Unlimited Text** | `TEXT` | `TEXT` | `LONGTEXT` | Large object storage (docs, JSON blobs, logs). |

### The Engineering Physics of Text

1.  **TOAST / Overflow Storage (PostgreSQL):**
    - By default, a row cannot span multiple 8KB pages.
    - If you store a 500KB log in a `TEXT` column, PostgreSQL **compresses** it (LZ) and then **moves it out of the main table** to a hidden TOAST table.
    - **Implication:** Selecting `SELECT * FROM logs` is fine. **BUT:** Indexing a `TEXT` column only indexes the first ~2KB. Searching inside a large text block requires a **Full-Text Search (GIN)** index, not a standard B-Tree.

2.  **Collation (The Silent Query Killer):**
    - Every text column has a **Collation** (e.g., `en_US.utf8`, `C`, `POSIX`).
    - `ORDER BY name` using `en_US.utf8` sorts "café" after "cafe" correctly but requires CPU-intensive Unicode tables.
    - `ORDER BY name COLLATE "C"` sorts by raw byte value (fastest, uses memcmp). Use `C` collation for UUIDs or internal keys where human language rules don't matter.

3.  **The VARCHAR(255) Anti-Pattern:**
    - Developers copy-paste `VARCHAR(255)` because it was historically the max for old MySQL indexes.
    - **Modern reality:** There is **zero** performance difference between `VARCHAR(50)` and `VARCHAR(255)` in PostgreSQL. The length constraint is a **business logic check**, not a storage optimization. Use `TEXT` + a `CHECK` constraint for clarity.

## 3. Date/Time Types
**The Nightmare of Time Zones and Daylight Saving**

### Core Types (PostgreSQL Canonical)

| Type | Size | Range | Resolution | Purpose |
| :--- | :--- | :--- | :--- | :--- |
| `DATE` | 4 bytes | 4713 BC - 5874897 AD | 1 day | Birthdays, Event Dates |
| `TIME` | 8 bytes | 00:00 - 24:00 | 1 microsecond | Opening hours (but not on a plane) |
| `TIMESTAMP` | 8 bytes | 4713 BC - 294276 AD | 1 microsecond | **"A point on the wall calendar."** |
| `TIMESTAMPTZ` | 8 bytes | Same | 1 microsecond | **"A specific instant in the universe."** |
| `INTERVAL` | 16 bytes | +/- 178M years | microsecond | Math results: `'1 day'::INTERVAL` |

### The Most Important 30 Seconds of Your Career: TIMESTAMP vs. TIMESTAMPTZ

- **`TIMESTAMP` (naive / local):** "The clock said 3:00 PM in the office in New York."
- **`TIMESTAMPTZ` (absolute / UTC):** "The moment the atomic clock in Colorado ticked 19:00 UTC."

**The Scenario:**
1. You store `TIMESTAMP '2026-04-16 15:00:00'`.
2. You move your database server from New York to London.
3. Suddenly, all your reports are wrong by 5 hours because the meaning of "15:00" changed.

**The Engineer's Rule:**
- **Always store `TIMESTAMPTZ`.** Always. It normalizes to UTC internally.
- **Only use `DATE` or naive `TIMESTAMP`** if the **physical location context is irrelevant and unchangeable** (e.g., "Independence Day is July 4th"). Even then, `TIMESTAMPTZ` is safer.

### Date/Time Arithmetic Traps
- **MySQL `TIMESTAMP`:** 32-bit. Overflows in **2038**. (Fixed in newer versions, but legacy tables exist). Use `DATETIME` in MySQL.
- **Adding Months:** `'2026-01-31'::DATE + INTERVAL '1 MONTH'`. What's the answer? Feb 28/29 or March 2? PostgreSQL returns **Feb 28** (last day of month logic). Know your DB's rules.

## 4. JSON & JSONB
**The Gateway Between Relational Rigidity and Schema Flexibility**

### The Fork in the Road: `JSON` vs. `JSONB`

| Feature | `JSON` (Text Storage) | `JSONB` (Binary Storage) |
| :--- | :--- | :--- |
| **Storage** | Exact text copy (whitespace preserved) | Binary decomposition (keys sorted) |
| **Write Speed** | Fast (just append string) | Slow (requires parsing) |
| **Read Speed** | **Slow** (parse JSON on every query) | **Fast** (pre-parsed, indexable) |
| **Indexing** | Not possible (only full text search) | **GIN Index** on keys/values |
| **Duplicate Keys** | Preserved (`{"a":1, "a":2}` keeps both) | Last key wins (`{"a": 2}`) |

**Verdict:** Use `JSONB` in PostgreSQL **unless** you are writing a log archive system where you just need to dump the exact HTTP request and rarely query inside it.

### Engineering Patterns for JSONB

1.  **The Mixed Schema Table:**
    ```sql
    CREATE TABLE events (
        id BIGINT PRIMARY KEY,
        payload JSONB,
        event_type TEXT GENERATED ALWAYS AS (payload->>'type') STORED
    );
    ```
    Store arbitrary event data, but extract a key column (`event_type`) for efficient partitioning and filtering *before* diving into the JSON.

2.  **The `@>` Operator (Containment):**
    ```sql
    -- Fast if you have a GIN index on payload
    SELECT * FROM events WHERE payload @> '{"user_id": 123, "action": "login"}';
    ```
    This checks if the JSON document *contains* that structure. It is the single most powerful query pattern in JSONB.

3.  **The Anti-Pattern:** **Updating a Nested JSON Key.**
    Updating `data -> 'user' -> 'email'` on 10,000 rows is **orders of magnitude slower** than updating a normal `users.email` column. JSONB is **Optimized for Read-Heavy Workloads**. If you find yourself doing bulk updates inside a JSON blob, you have chosen the wrong data model.

## 5. Arrays
**When the 1NF Police Aren't Looking**

### Core Concept
PostgreSQL allows columns to store lists of native types: `INTEGER[]`, `TEXT[]`.

### When to Use Arrays (Legitimately)

1.  **Denormalized Tags/Labels:**
    `tags TEXT[] = ['postgres', 'database', 'tutorial']`.
    **Why it's better than 1NF:** A `post_tags` junction table requires 3 I/O operations to fetch tags for 10 posts (1 post scan + 2 index seeks per post). An array column loads all tags in the **same 8KB page** as the post. **Read performance gain: 10x.**

2.  **Order Preservation:**
    A junction table (`post_tags`) is inherently **unordered** unless you add a `position` column.
    An array (`['intro', 'body', 'conclusion']`) intrinsically preserves the sequence of items.

### The "Don't Do It" List
- **Never search `WHERE array_column @> ARRAY['value']` without a **GIN Index**. Without it, the DB scans the entire table and unnests every array.
- **Never use Arrays for Mutable Relationships.** If tags change frequently and you need to query "All posts with tag X", a junction table with a B-Tree index on `(tag_id, post_id)` will **write faster and read faster** than updating array columns.

## 6. Enums
**The Politically Contentious Data Type**

### Definition
A static, ordered list of allowed string values.
`CREATE TYPE mood AS ENUM ('sad', 'ok', 'happy');`

### The Good
- **Storage Efficiency:** Stored as a **4-byte integer** internally (the index in the enum list).
- **Type Safety:** Impossible to insert `'hapy'` (typo).
- **Sort Order:** Sorts by the *order of definition*, not alphabetically. `'happy' > 'sad'` is true because 3 > 1.

### The Bad (The Reason Some Engineers Ban Enums)
- **Immutability Nightmare:** You cannot drop a value from an `ENUM`. You cannot rename a value without a full table rewrite in some DBs.
- **Migration Pain:** Adding a new value requires `ALTER TYPE ... ADD VALUE`. In PostgreSQL prior to v12, this took a **ShareLock** (blocking reads) unless you added it as the *last* value.
- **Cross-DB Portability:** Enums are **not standard SQL**. Moving from PostgreSQL to MySQL involves significant schema translation.

### The Practical Alternative
**Use a Lookup Table + Foreign Key Constraint.**
```sql
CREATE TABLE mood_types (id SMALLINT PRIMARY KEY, label TEXT UNIQUE);
-- Pros: Easy to add/rename/retire values with simple INSERT/UPDATE.
-- Cons: Joins required (though small tables are often cached in memory).
```

### The Verdict on Enums
Use them **only if**:
1. The list of values is **truly immutable** (e.g., Days of Week, Suits in a Deck of Cards).
2. You are staying on a single database vendor forever.
3. You care deeply about saving 20 bytes per row in a 1-billion-row table.
